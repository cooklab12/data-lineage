Ataccama ONE: Step-by-Step Guide from Data Catalog to DQ Rule Setup

Step 1: Understand Data Catalog and its role

A Data Catalog is a centralized inventory of your data assets. It provides metadata—data about your data—which helps you understand, find, and trust your data. In the context of Data Quality (DQ), the Data Catalog is crucial because it provides the necessary information and starting point for defining and applying DQ rules.

In Ataccama ONE, you typically start by connecting to your various data sources. This process brings metadata about your tables, columns (attributes), and other data assets into the Catalog. Often, this is followed by data profiling, which analyzes the content and structure of your data, enriching the catalog with statistics, detected patterns, and potential quality issues. This enriched catalog then serves as the foundation for creating targeted DQ rules. For instance, you might browse the catalog, identify an attribute with suspect data (perhaps from profiling insights), and then initiate rule creation directly from that attribute.

Step 2: Navigate to the Data Catalog in Ataccama ONE

Once you are in the Ataccama ONE environment, you'll typically find the Data Catalog through a main navigation menu. Look for options like 'Data Catalog,' 'Catalog Items,' or 'Browse Catalog.'

After accessing the catalog, you can:

  • Browse: Navigate through your connected data sources, schemas, and tables. This is useful when you have a general idea of where the data resides.
  • Search: Use the search functionality (often indicated by a search bar) to find specific catalog items (tables, views, etc.) or attributes by name or other metadata properties.

Once you've located the relevant catalog item (e.g., a specific table like 'CUSTOMERS' or 'PRODUCTS'), you will select it to view its details. Within the details screen for that catalog item, you'll typically find tabs such as 'Overview,' 'Attributes,' 'Profile & DQ insights,' or 'Data Quality.' These tabs allow you to explore the item's metadata, see its attributes (columns), review profiling results, and manage associated DQ rules. To set up a rule for a specific piece of data, you'll first need to identify the attribute within this view.

Step 3: Initiate DQ Rule Creation from Data Catalog (Quick Method)

Once you've identified the specific attribute in the Data Catalog that requires a data quality rule, Ataccama ONE provides a 'Quick DQ rule creation from attribute' workflow. This method is optimized for speed and convenience when you already have a good idea of the rule logic.

Here's how to initiate it:

  1. Locate the Attribute: Navigate to the catalog item (e.g., a table) and find the attribute in its listing. This listing is often found on tabs like 'Overview,' 'Attributes,' or 'Profile & DQ insights.'
  2. Access Rule Creation:
    • Typically, you'll find a three-dots menu (⋮) or a similar context menu next to the attribute name. Click on this menu.
    • From the options that appear, select 'Create DQ Rule' (or a similarly phrased option).

(Alternative: Some parts of the interface might have an 'Add Rule' button that then allows you to 'Create Rule' if a suitable one doesn't already exist.)

This action will open the 'Create DQ Rule' dialog, pre-configured with the selected attribute as the primary input for your new rule.

Step 4: Define Basic Rule Details (Quick Creation Workflow)

Upon initiating the 'Quick DQ Rule' creation, a dialog window will appear. This dialog is designed to streamline the process, and Ataccama ONE helpfully pre-fills some basic details for your new rule:

  • Input Attribute: The attribute you selected in the previous step (e.g., 'EMAIL_ADDRESS') will automatically be set as the primary input for the rule. Its data type (e.g., String, Integer) will also be pre-filled.
  • Rule Dimension: A default Data Quality Dimension is typically pre-selected. Often, this is 'Validity,' which is a common starting point for many rules (verifying data format, content, etc.). You can usually change this dimension if needed (e.g., to Completeness, Uniqueness, Accuracy).

The dialog will then present you with an area to define the actual logic of your rule. If it's the very first time or a very new draft, this area might be empty, prompting you to 'Add condition.' Otherwise, a basic rule structure might already be visible.

At this stage, you confirm that the pre-filled input attribute is correct and that the chosen DQ dimension aligns with the kind of check you want to perform.

Step 5: Specify Rule Logic

This is where you define what the rule will actually check. After confirming the basic details, you'll focus on the 'Rule Logic' section of the 'Create DQ Rule' dialog.

  1. Adding Conditions: You build the rule by adding one or more conditions. Each condition evaluates the input attribute(s) against specific criteria.
  2. Methods for Defining Logic:
    • Condition Builder: This is a user-friendly interface that allows you to construct rule logic by selecting options from dropdown menus and filling in values. You typically:
      • Confirm the input attribute.
      • Choose an operator or function (e.g., 'is empty,' 'contains,' 'matches mask,' 'is greater than').
      • Provide any necessary values for the operator (e.g., the specific text to contain, the regular expression for the mask, the number for comparison).
      • Specify what the outcome should be if the condition is met (e.g., if 'EMAIL_ADDRESS' does not contain '@', then the result is 'Invalid').
    • Advanced Expression (Optional): For more complex scenarios or users comfortable with Ataccama's expression language (ONE Expressions), there's often an option to switch to an 'Advanced Expression' mode. This allows you to write custom code to define the rule's logic, offering greater flexibility.

Simple Example (using Condition Builder):

Let's say we are creating a rule for an 'EMAIL_ADDRESS' attribute, and we want to check if it's not empty.

  • Input Attribute: EMAIL_ADDRESS (pre-filled)
  • Condition: You would add a condition like:
    • Input: EMAIL_ADDRESS
    • Operator: Is empty
    • Result if true: Invalid (Meaning, if the email address is empty, it's considered invalid according to this rule).
  • You might also define a corresponding 'Valid' result if the condition Is not empty is true.

You can often chain multiple conditions together using AND/OR logic to build more sophisticated rules.

Step 6: Test the Rule

Before saving and applying a new Data Quality rule, it's crucial to test it to ensure it behaves as expected. Ataccama ONE provides a testing facility directly within the rule creation dialog.

  1. Access Testing Interface: Look for a 'Test Rule' button or tab. Clicking this will typically open a test panel.
  2. Input Test Data:
    • The test interface usually includes a table where your rule's input attribute(s) are shown as columns.
    • This table might be pre-filled with sample data from the attribute if data samples are available in the catalog and you have permission to view them.
    • You can manually enter various test values into these columns. It's important to test with data that should pass the rule, data that should fail, edge cases, and different formats if applicable.
    • You can often add or remove rows in this test table to cover multiple scenarios.
  3. Observe Results: As you input or modify test data, the rule is evaluated in real-time (or by clicking a 'Run Test' button). The outcome for each test row (e.g., 'Valid,' 'Invalid,' 'Accurate,' 'Not Accurate,' plus any score if defined) is displayed.
  4. Iterate if Necessary: If the test results are not what you expect, you can go back to the rule logic definition, make adjustments, and then re-test until you are confident the rule is correct.

This iterative testing process helps catch errors early and ensures your DQ rules are effective.

Step 7: Provide General Information and Publish (Quick Creation)

Once you've defined and tested your rule's logic, you'll proceed to the final step in the quick creation workflow: providing general information and publishing the rule.

  1. Navigate to General Information: After testing, select 'Next step' (or a similar button) in the 'Create DQ Rule' dialog. This will take you to a screen where you provide metadata for your new rule.
  2. Fill in Rule Details:
    • Name: Give your rule a clear and descriptive name (e.g., 'Email Address Format Check', 'Order Quantity Must Be Positive'). This is mandatory.
    • Description (Optional but Recommended): Provide a more detailed explanation of what the rule does, its purpose, and perhaps how to interpret its results.
    • Stewardship (Recommended): Assign a user group responsible for this rule.
    • Share with (Optional): Specify which user groups or individual users should have access to see or use this rule.
  3. Publish and Apply:
    • After filling in the general information, you'll select an option like 'Publish and apply'.
    • The exact wording might vary (e.g., 'Publish', 'Request publish') depending on permissions and review workflows.
  4. Confirmation: A confirmation message will typically appear.
  5. Activate by Publishing Catalog Item: For the rule to be used in DQ evaluations, the catalog item (table/asset) itself usually needs to be published.

Step 8: Alternative - Create DQ Evaluation Rule in the Rule Library

While the 'Quick DQ rule creation from attribute' method is convenient, Ataccama ONE also provides a more comprehensive way to create rules: directly within the Rule Library. This method is preferred when:

  • You need access to more advanced rule creation options (e.g., defining parameters, variables, complex scoring).
  • You want to create a reusable rule not immediately tied to a specific attribute.
  • You are developing a complex rule requiring more detailed configuration.

Navigation and Initial Creation:

  1. Navigate to the Rule Library: Go to 'Data Quality' > 'Rules'.
  2. Create New Rule: Click the 'Create' button.
  3. Provide Initial Information:
    • Name: Unique name for your rule.
    • Rule Type (DQ Dimension): Explicitly select the DQ dimension (e.g., Validity, Uniqueness).
    • Description, Rule Definition Source, Stewardship.
  4. Save Draft: Click 'Save'. The system will then indicate that the rule's logic needs to be defined.

You will then proceed to the 'Implementation' tab for this new rule.

Step 9: Rule Implementation in the Library (Advanced Options)

After creating a draft rule in the Rule Library, you'll define its core logic and advanced characteristics in its 'Implementation' tab.

1. Define Inputs, Parameters, and Variables:

  • Input Attributes: Explicitly 'Add attribute', provide a name and select its data type. You can associate Terms.
  • Parameters (Optional): Make rules reusable (e.g., `Age > [MinimumAgeParam]`). Select 'Add parameter,' provide a name, and data type. Values are provided when applying the rule.
  • Variables (Optional): Perform transformations on inputs before logic is applied (e.g., convert to uppercase). Select 'Add variable,' name it, choose the base input attribute, and apply transformations.

2. Define Rule Logic:

  • Rule Type Context: You've already selected the DQ Dimension.
  • Standard, Aggregation, or Component Rule: Specify the rule's nature.
  • Condition Builder vs. Advanced Expression: Use the interactive builder or write custom logic with ONE Expressions.
  • Define Outcomes: For each condition, define the result (e.g., 'Valid', 'Invalid') and optionally an explanation.

3. Assigning Scores (Optional):

  • Assign a numerical score to different outcomes, representing severity. Useful in Monitoring Projects.

Once implemented, the rule is ready for testing and publishing.

Step 10: Publish Rule from Library

Once you have fully defined your Data Quality rule in the Rule Library and tested it, the next step is to make it official by publishing it.

  1. Locate the Publish Action: Within the rule's interface in the Rule Library, find a 'Publish' button or action.
  2. Review and Confirm: The exact action might depend on permissions and workflows ('Publish', 'Request Publish', or 'Submit for Review').
  3. Impact of Publishing: Finalizes the rule's definition, makes it available for others to discover and apply, and usable in Monitoring Projects.

After publishing, the rule is a standalone, reusable asset.

Step 11: Apply the Rule

Whether you created your rule via the 'Quick Creation' method (where it's often automatically applied to the source attribute) or created it in the Rule Library, you need to ensure it's correctly associated with the data it's meant to evaluate.

Applying Rules from the Library (or to other attributes):

  1. Navigate to the Catalog Item: Go to the Data Catalog and find the specific catalog item (table, view) and then the attribute(s) to which you want to apply a rule.
  2. Add Rule to Attribute:
    • On the attribute's detail screen or context menu, look for an option like **'Add Rule'** or **'Manage Rules'**.
    • This will typically open an 'Add Rule' dialog.
    • Here, you can search or browse the Rule Library for the published rule you want to apply. Select your rule.
  3. Configure Parameters (if any): If the rule uses parameters (e.g., `[MinimumAgeParam]`), you will be prompted to provide values for these parameters in the context of this specific attribute. For example, for one attribute, `MinimumAgeParam` might be 18, and for another, it might be 21.
  4. Add Rule to Term (Alternative): You can also associate DQ rules with Business Terms in your Glossary. When a term is then linked to attributes in the catalog, these associated rules can be automatically suggested or applied, promoting consistency.

Publish Catalog Item Changes:

Crucially, after applying a new rule (or changing rule associations) to a catalog item, you usually need to **publish the catalog item itself**. This action finalizes the changes to the catalog item's metadata, including its DQ rule configuration, making the rules active for subsequent DQ evaluations and monitoring.
